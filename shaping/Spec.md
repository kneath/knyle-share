# Knyle Share Spec

## Summary

Knyle Share is a Rails application for publishing local files or folders to the web at `https://share.warpspire.com/<bundle-slug>`.

The product is optimized for fast, low-friction sharing from the command line. A bundle can be public or protected. Protected bundles can be opened either with a shared password or with a signed URL parameter that expires after a chosen amount of time.

The application includes:

- A Rails web app hosted on Render
- A private admin area for bundle management and analytics
- An HTTP API for uploads and bundle management
- A local CLI for creating and uploading bundles
- An LLM skill that delegates to the CLI instead of reimplementing upload logic
- A deployment path that is easy for other people to run with a Render account and an AWS account

## Product Goals

- Share a local file or folder to a stable URL in one short terminal flow
- Support both human-friendly password sharing and expiring link sharing
- Make common bundle types pleasant to consume without extra setup
- Keep admin management minimal and obvious
- Preserve enough analytics to answer "did anyone besides me view this?"
- Keep self-hosting setup cheap and approachable for a small personal deployment

## Non-Goals

- Multi-user accounts or team collaboration
- Rich content editing in the web app
- Full document collaboration or commenting
- High-scale public hosting or CDN-grade performance in v1
- Public search, discovery, or indexing of bundles

## Users

### Admin

The configured admin user uploads bundles, manages access, generates expiring links, and reviews basic analytics.

### Viewer

A viewer opens a shared bundle URL. Depending on bundle settings, they either see the content immediately, unlock it with a password, or arrive via a signed expiring link.

### Automation

The CLI and LLM skill act as trusted clients for upload workflows.

## Core Concepts

### Bundle

A bundle is the top-level published unit. It has:

- A unique slug used in the URL
- A source type of `directory` or `file`
- A presentation type inferred from its contents
- A visibility status of `active`, `disabled`, or `deleted`
- An access mode of `public` or `protected`
- Password access when protected
- Signed-link access when protected
- Uploaded contents and metadata

### Presentation Types

The server classifies a bundle into one of these behaviors:

- `static_site`: a directory with a root `index.html`, served as a website
- `markdown_document`: a single Markdown file rendered to HTML with raw and download actions
- `single_download`: a single non-Markdown file rendered as a landing page with a download button
- `file_listing`: a directory without a root `index.html`, rendered as a simple file browser

## URL Structure

- Public bundle URL: `https://share.warpspire.com/<bundle-slug>`
- Signed access form: `https://share.warpspire.com/<bundle-slug>?access=<token>`

Reserved namespaces must not be claimable as bundle slugs. At minimum:

- `api`
- `assets`
- `rails`
- `up`
- `health`

## Bundle Lifecycle

1. Admin runs `knyle-share <path>` locally
2. CLI inspects the path and proposes a bundle slug
3. CLI validates the slug with the server before upload
4. If the slug already exists, CLI asks whether the existing bundle should be replaced unless replacement was explicitly requested with a flag
5. CLI asks whether the bundle should be public or protected
6. If protected, CLI offers a generated memorable password or accepts a custom password
7. CLI uploads the source to the API
8. Server stores the upload, classifies the bundle, and publishes it
9. CLI prints the final URL and any access details
10. After creation, CLI offers to copy the share URL or generate and copy an expiring link
11. Admin can later disable, re-enable, or delete the bundle from the admin area

## Access Control

### Public Bundles

Anyone with the URL can view the bundle.

### Protected Bundles

Protected bundles support both of these access paths:

- Shared password entry
- Signed URL parameter with explicit expiration

This means a single protected bundle may be opened either by password or by an expiring token. The expiring token bypasses the password screen until the token expires.

### Password Flow

- The password is stored as a secure digest, never in plaintext
- After a correct password entry, the viewer receives a bundle-scoped session
- Default access duration is 24 hours
- After expiry, the viewer must re-enter the password
- Password attempts should be rate-limited

### Signed URL Flow

- Expiring URLs are generated by the server, not the client
- The token embeds the bundle identity and expiry time and is signed by the app
- The token is passed in a URL parameter
- Expiry is configurable when the link is generated
- An expired or tampered token renders the password screen or an expired-link message

## Bundle Rendering Rules

### Static Website

If the uploaded bundle is a directory with `index.html` at its root:

- Serve the site under `/<bundle-slug>`
- Resolve relative asset paths within that bundle
- Preserve nested files and directories
- Apply bundle auth checks to every asset request

### Markdown Document

If the uploaded bundle is a single Markdown file:

- Render it to HTML using a standard site stylesheet
- Show the document title if one can be inferred
- Include actions to download the original file and view the raw source

### Single File Download

If the uploaded bundle is a single non-Markdown file:

- Render a simple landing page with file name, size, and type if known
- Show a prominent download button

### File Listing

If the uploaded bundle is a directory without `index.html`:

- Render a clean file listing with nested navigation
- Allow downloading individual files
- Optionally allow downloading the original uploaded archive if retained

## Admin Area

The admin area is private and only used by the configured admin user.

### Bundle List

The index view should show:

- Bundle name and slug
- Presentation type
- Access mode
- Status
- Created at and last updated at
- Last viewed at
- Total views
- Unique viewers for protected bundles

### Bundle Actions

The admin must be able to:

- View bundle details
- Enable a disabled bundle
- Disable an active bundle
- Delete a bundle
- Change bundle password
- Generate signed links with chosen expiration

### Admin Authentication

The admin area should use GitHub OAuth.

For v1:

- On first startup, if no admin user has been configured yet, the app must first pass setup validation before allowing an admin to be claimed
- After setup validation passes, the first successful GitHub OAuth login becomes the admin user
- After initial bootstrap, only that configured admin user may log in
- Store the configured admin GitHub identity in the application database
- Require admin authentication for all admin routes and signed-link generation screens

### First-Run Bootstrap

The application should support a one-time admin bootstrap flow.

- A fresh installation starts with no configured admin user
- Before the admin can be claimed, the app runs a self-validation step that confirms required configuration is present and usable
- The first successful GitHub OAuth login after a successful validation claims the admin role
- After that first claim, later GitHub logins must match the stored admin identity
- The UI should make it clear when the application is in unclaimed bootstrap mode
- The UI should make it clear when setup validation has failed and what must be fixed
- This bootstrap flow should only apply to the admin origin, never to public bundle URLs

### Setup Validation

Before first-run admin setup, the app should validate that the installation is actually usable.

- Validate that all required environment variables are present
- Validate that the application database is reachable and migrations have been applied
- Validate that S3 configuration is present
- Validate that the configured S3 bucket is reachable with the supplied credentials
- Validate that the app can perform a small write, read, and delete cycle against S3
- Refuse admin bootstrap until validation passes
- Present a concise checklist of passing and failing validation items in the first-run UI
- Allow the validation check to be re-run after configuration changes

### Analytics

Analytics are intentionally simple. The system should capture:

- Total bundle views
- Last viewed at
- Whether any non-admin viewer has accessed the bundle
- Approximate unique viewer count for protected bundles

For v1, unique viewers should be approximated by authenticated viewer sessions, not a full analytics platform.

Admin-authenticated requests should be excluded from viewer analytics.

Analytics stay attached to the slug across bundle replacements. Replacing bundle contents does not reset historical view counts for that slug.

## CLI

The CLI is the primary publishing interface.

### Core Command

The main command is:

```bash
knyle-share <path>
```

Examples:

```bash
knyle-share .
knyle-share ./app
knyle-share "./Summer in the Sierra.md"
knyle-share ./Blueprints.zip
```

### CLI Responsibilities

- Inspect the provided path
- Infer a default slug from the file or directory name
- Ask for confirmation before uploading
- Validate slug availability with the server
- Ask for confirmation before replacing an existing slug
- Ask whether the bundle should be public or protected
- Offer a generated memorable password for protected bundles
- Accept a custom password instead
- Upload the bundle
- Print the resulting URL and access details
- Offer to copy the share URL after upload
- For protected bundles, offer to generate and copy an expiring link after upload

### Generated Password Requirements

Generated passwords should be:

- Human-memorable
- Easy to say aloud
- Easy to write on paper
- Lower friction than a random symbol-heavy password
- Consist of exactly three words
- Use only lowercase letters
- Separate words with spaces
- Avoid numbers, dashes, and symbols

Required format: three lowercase words separated by spaces, such as `river maple lantern`.

### CLI Modes

The CLI should support:

- Interactive mode for normal human use
- Non-interactive flags for scripting later
- A machine-readable output mode for LLM tooling

### Replacement Behavior

- Re-uploading to an existing slug replaces the current published bundle
- Interactive CLI usage must confirm replacement before upload
- A replacement flag skips the interactive replacement confirmation
- Non-interactive replacement should require an explicit replacement flag
- Replacement should publish atomically so a viewer never sees a half-updated bundle
- Existing analytics remain attached to the slug after replacement
- Existing password sessions remain valid after replacement until they naturally expire
- Existing signed links remain valid after replacement until they naturally expire

### Post-Upload UX

After a successful upload, the CLI should present the most useful next actions.

- Always show the final share URL
- Offer to copy the share URL to the clipboard when clipboard access is available
- For protected bundles, offer to generate an expiring link and copy it to the clipboard
- If clipboard access is unavailable, print the value clearly instead of copying it

### Expiring Link Presets

The default expiration presets should be:

- 1 day
- 1 week
- 1 month

## LLM Skill

The skill should call the CLI instead of embedding upload logic in the prompt.

The skill must support prompts such as:

- "Share this bundle"
- "Share the `app` folder as a bundle"

The skill should:

- Resolve the target path from the working directory or explicit user request
- Ask for the same confirmations as the CLI when needed
- Reuse the CLI's validation and upload flow

## API

The Rails app exposes a private API used by the CLI.

### Required Endpoints

- Check slug availability
- Create bundle upload
- Finalize and publish bundle
- Fetch bundle metadata for CLI confirmation output
- Generate signed access links
- Replace an existing bundle when replacement is explicitly requested

The exact route shape can be designed during implementation, but the API should remain small and purpose-built.

## Data Model

The exact schema can change, but the initial model set should likely include:

### Bundle

- `slug`
- `source_kind`
- `presentation_kind`
- `status`
- `access_mode`
- `password_digest`
- `password_session_ttl`
- `title`
- `description` optional
- `root_path` or manifest reference
- `byte_size`
- `created_at`
- `updated_at`
- `last_viewed_at`

### BundleAsset

- `bundle_id`
- `path`
- `content_type`
- `byte_size`
- `storage_key`
- `checksum`

### BundleView

- `bundle_id`
- `access_method`
- `viewer_session_id` optional

- `bundle_id`

## Storage and Hosting

### Deployment Goal

The default production deployment should be easy for another person to set up with:

- One Render web service
- One Render persistent disk
- One AWS S3 bucket

The repository should include a straightforward Render deployment path, ideally via `render.yaml`, plus concise setup documentation for the required AWS S3 configuration.

The deployment path should also support the first-run validation flow so a new operator can confirm the app is configured correctly before claiming the admin account.

### Application Hosting

- Rails app hosted on Render
- Default production database is SQLite on a Render persistent disk
- Development also uses SQLite by default
- Postgres is a supported optional backend for advanced deployments that need horizontal scaling, zero-downtime deploys, or a separate managed database

### File Storage

Bundle files should not live only on the Render filesystem. Use object storage for durability.

- AWS S3 is the documented default object storage provider
- The storage integration should remain S3-compatible so alternative providers are possible later
- Rails stores metadata in the application database and file bytes in the bucket
- First-run setup validation must verify that S3 credentials, bucket, and object operations are working before admin bootstrap is allowed

### Upload Size

There is no product-level upload size limit in v1.

The implementation should therefore:

- Avoid loading full bundle contents into memory
- Prefer streamed upload and extraction paths
- Keep the app compatible with large files and large directories within the practical limits of the hosting and storage providers

## Security Constraints

### Admin Origin Separation

Uploaded static sites may contain arbitrary HTML, CSS, and JavaScript. Serving admin pages from the same origin as uploaded bundles creates avoidable same-origin risk.

- Public bundles stay at `share.warpspire.com/<bundle-slug>`
- Admin area lives on a separate origin such as `admin.share.warpspire.com`

### Other Security Requirements

- Use HTTPS only
- Hash passwords with bcrypt
- Sign expiring links with a server secret
- Rate-limit password attempts
- Sanitize Markdown rendering
- Prevent path traversal in uploaded archives or directories
- Treat bundle content as untrusted

## Operational Constraints

- Bundle publishing should feel quick, but correctness is more important than instant availability
- Disabled bundles should return a clear unavailable page
- Deleted bundles should not remain publicly accessible
- The app should reserve room for future bundle replacement/versioning, but v1 can ship with a simpler model
- Replacements should reuse the same public URL
- The default SQLite deployment targets a single Render service instance
- The default Render deployment can tolerate brief downtime during deploys in exchange for lower cost and simpler setup

## Confirmed v1 Decisions

The build should assume:

- Protected bundles support both password access and signed-link access
- Password sessions last 24 hours by default
- Directories without `index.html` render as file listings
- Re-uploading the same slug replaces the existing bundle after confirmation, or immediately when a replacement flag is supplied
- Replacement preserves analytics on the slug and does not invalidate existing password sessions or signed links
- Admin runs on a separate origin from public bundles
- Admin authentication uses GitHub OAuth
- The application performs setup validation before first-run admin bootstrap, including environment, database, and S3 checks
- The application sets its admin user during the first successful GitHub OAuth login after setup validation passes on a fresh installation
- The default production deployment uses SQLite on a Render persistent disk
- AWS S3 is the documented default object storage backend, while the integration remains S3-compatible
- Postgres remains an optional upgrade path for more operationally demanding deployments
- The CLI offers to copy the share URL or generate and copy an expiring link after bundle creation
- Expiring link presets are `1 day`, `1 week`, and `1 month`
- The application does not impose a product-level upload size limit
